# compliance-check-service

Regulatory compliance gatekeeper for CredityFlow loan contracts. Every contract generated by the platform passes through this service before it can be digitally signed. The service evaluates contracts against BACEN resolutions and consumer protection law (CDC), producing a clear pass/fail verdict along with detailed violation reports when applicable.

Owned by **team-contracts** | Domain: **Contract** | Service ID: `svc-28`

## Getting Started

### Prerequisites

- JDK 17+
- Docker & Docker Compose (for local PostgreSQL)
- Kafka broker (or use the dev Docker Compose stack)

### Running Locally

```bash
# Start dependencies
docker compose -f infra/docker-compose.yml up -d

# Run the service
./gradlew bootRun --args='--spring.profiles.active=local'
```

The service starts on port **8080** by default. Health check is available at `/actuator/health`.

### Running Tests

```bash
./gradlew test          # unit tests
./gradlew integrationTest  # requires Docker
```

## Architecture

The service follows a hexagonal architecture pattern. The compliance rule engine sits at the core, surrounded by adapters for Kafka event consumption, REST API exposure, and PostgreSQL persistence.

Key components:
- **Rule Engine** — evaluates a contract against all applicable `ComplianceRule` records.
- **Kafka Consumer** — listens for `contract_generated` events and triggers the engine.
- **REST Controller** — exposes endpoints for on-demand checks and rule listing.
- **Audit Repository** — persists every `ComplianceResult` for regulatory audit purposes.

For a deeper dive, see [architecture.md](./architecture.md).

## API Reference

### POST /api/v1/compliance/check

Run compliance validation against a contract payload.

**Request body:**
```json
{
  "contract_id": "uuid",
  "product_type": "VEHICLE_LOAN",
  "terms": { "...contract terms..." }
}
```

**Response (200):**
```json
{
  "contract_id": "uuid",
  "verdict": "PASS",
  "violations": [],
  "checked_at": "2026-03-29T10:00:00Z"
}
```

A `FAIL` verdict includes a `violations` array with rule code, description, and severity for each issue found.

### GET /api/v1/compliance/rules

Returns the list of active compliance rules. Supports query parameters `product_type` and `regulation_source` for filtering.

## Configuration

Key environment variables:

| Variable | Description | Default |
|---|---|---|
| `SPRING_DATASOURCE_URL` | PostgreSQL JDBC URL | `jdbc:postgresql://localhost:5432/contract_db` |
| `KAFKA_BOOTSTRAP_SERVERS` | Kafka broker addresses | `localhost:9092` |
| `KAFKA_CONSUMER_GROUP` | Consumer group ID | `compliance-check-cg` |
| `COMPLIANCE_RULES_CACHE_TTL` | Rule cache TTL in seconds | `300` |

Spring profiles: `local`, `staging`, `production`.

## Deployment

Deployed as a container on EKS via Helm chart. The CI/CD pipeline runs on GitHub Actions:

1. Push to `main` triggers build + unit tests.
2. Merge to `release/*` deploys to staging.
3. Production deploy requires manual approval.

Resource defaults: 512Mi memory request, 1Gi limit. CPU: 250m request, 500m limit. Two replicas minimum in production.

## Monitoring

Prometheus metrics are exposed at `/actuator/prometheus`. Primary dashboards in Grafana:

- **Compliance Check Overview** — pass/fail rates, latency percentiles, top violated rules.
- **Rule Engine Health** — active rule count, evaluation durations.

Alerts are configured in PagerDuty for high failure rates (>30% over 15 min) and elevated p99 latency (>5s).

## Known Issues

- Regulatory rule updates are manual. There is no automated feed from BACEN, so new resolutions require a developer to encode and deploy updated rules.
- The rule cache (default 5 min TTL) means freshly deployed rules may take up to 5 minutes to take effect across all pods.
- No circuit breaker on the database layer; a PostgreSQL outage will cause cascading failures on the Kafka consumer.
